# CODING STANDARDS

Версия документа: 1.0

---

# Назначение

Данный документ определяет единый стиль разработки проекта ADOKVA.

Все новые компоненты, функции и модули должны соответствовать этим правилам.

Главная цель — сделать проект единообразным, читаемым и удобным для развития в течение многих лет.

---

# Основной принцип

Каждый новый файл должен выглядеть так, словно его писал один и тот же разработчик.

Стиль проекта важнее личных предпочтений.

---

# Язык

Исходный код:

Английский язык.

Документация:

Русский язык.

Комментарии:

Русский или английский только при необходимости.

---

# Структура компонента

Каждый React-компонент должен иметь одинаковый порядок.

```
Imports

Types

Constants

Component

Hooks

Functions

Return

Export
```

---

# Импорты

Всегда группируются.

Сначала:

```
React
```

Затем:

```
Next.js
```

Потом:

```
Three.js
```

Потом:

```
Внешние библиотеки
```

Потом:

```
Локальные компоненты
```

Потом:

```
Типы
```

Потом:

```
Utils
```

---

# Компоненты

Название файла:

```
Planet.tsx
```

Название компонента:

```
function Planet()
```

или

```
const Planet = () => {}
```

Используется один стиль во всём проекте.

---

# Размер компонентов

Желательно:

до 250 строк.

Если компонент становится слишком большим —

разделить.

---

# Props

Всегда типизируются.

Пример:

```
type PlanetProps = {

    radius: number;

    texture: string;

}
```

Использование `any` запрещено.

---

# Константы

Все повторяющиеся значения выносятся.

Например:

```
PLANET_RADIUS

CAMERA_SPEED

SUN_POSITION

EARTH_SCALE
```

Никаких магических чисел.

---

# Типы

Все общие типы располагаются в папке:

```
types/
```

Запрещается дублировать одинаковые интерфейсы.

---

# Данные

Все данные располагаются отдельно.

Например:

```
data/

worldInformation.ts

planets.ts

countries.ts
```

Компоненты не должны хранить большие массивы данных.

---

# Логика

Вычисления выносятся в:

```
lib/

или

utils/
```

JSX должен содержать минимум вычислений.

---

# Hooks

Каждый hook отвечает только за одну задачу.

Например:

```
useCamera()

useOrbit()

useSelection()

useAnimation()
```

---

# JSX

Избегать длинных вложенных конструкций.

Если JSX становится трудно читать —

разделить компонент.

---

# useFrame

Внутри useFrame должна находиться только логика обновления анимации.

Запрещается выполнять тяжёлые вычисления.

---

# useMemo

Использовать только там, где действительно есть выигрыш.

Не применять без необходимости.

---

# useEffect

Использовать только для побочных эффектов.

Не применять как замену обычной логике.

---

# Three.js

Каждый космический объект располагается в отдельном компоненте.

Например:

```
Earth

Moon

Sun

Mercury

Mars

Venus
```

Нельзя объединять несколько планет в один компонент.

---

# Камера

Управление камерой должно быть централизовано.

Все перелёты выполняются через единую систему.

---

# Материалы

Настройки материалов должны быть максимально похожими.

Использовать единый стиль освещения.

---

# Анимации

Каждая анимация должна быть независимой.

Остановка одной системы не должна ломать остальные.

---

# CSS

Использовать единый стиль оформления.

Избегать случайных значений.

При необходимости использовать константы.

---

# Именование

Компоненты:

```
PascalCase
```

Переменные:

```
camelCase
```

Константы:

```
UPPER_CASE
```

Типы:

```
PascalCase
```

---

# Функции

Название функции должно объяснять действие.

Правильно:

```
moveCamera()

focusPlanet()

calculateOrbit()
```

Неправильно:

```
go()

run()

doSomething()
```

---

# Комментарии

Код должен быть понятен без большого количества комментариев.

Комментарии писать только там, где логика действительно сложная.

---

# Документация

После каждого крупного этапа обновляются:

- CURRENT_STATE.md
- CHANGELOG.md
- PROGRESS.md
- DECISIONS.md
- SESSION_SUMMARY.md

---

# Главное правило

Перед написанием нового кода необходимо задать себе вопрос:

> Можно ли через два года понять этот код за пять минут?

Если ответ отрицательный —

код необходимо упростить.
