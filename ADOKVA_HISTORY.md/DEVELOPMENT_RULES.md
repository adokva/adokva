# Назначение

Этот документ содержит правила разработки проекта ADOKVA.

Все изменения проекта должны соответствовать данным правилам.

---

# 1. Главное правило

НЕ ломать то, что уже работает.

Рабочий компонент считается стабильным.

Любое изменение должно быть минимальным.

---

# 2. Правило «ЙЙ»

Если пользователь пишет:

йй

это означает:

• присылать только полный файл;

• всегда указывать точный путь;

• один файл за один ответ;

• никаких кусочных изменений;

• никаких "найди строку";

• пользователь делает:

Ctrl+A

Ctrl+V

Ctrl+S

после чего проверяет:

Problems = 0

Только после этого разрешается переходить к следующему файлу.

---

# 3. Проверка после каждого изменения

После сохранения обязательно проверить:

Problems = 0

Если появились ошибки —

не переходить к следующему файлу.

---

# 4. Git

Перед коммитом всегда выполнить:

git status

После проверки проекта:

git add ...

git commit

Коммит запрещён,
если проект не проверен.

---

# 5. Большие файлы

Никогда полностью не переписывать большие компоненты.

Особенно:

Globe.tsx

FlightCamera.tsx

WorldCameraController.tsx

CanvasScene.tsx

SearchPanel.tsx

services/search.ts

Сначала анализ.

Потом минимальное изменение.

---

# 6. Архитектура

Каждый компонент должен иметь одну ответственность.

Не смешивать:

• UI

• Camera

• Search

• Earth

• Moon

• Sun

• World switching

---

# 7. Камера

FlightCamera

отвечает только за кинематографические перелёты.

WorldCameraController

отвечает только за переходы между Землёй,
Луной
и Солнцем.

FocusOrbitControls

отвечает только за управление пользователя.

CameraIntro

отвечает только за стартовое интро.

---

# 8. Анализ

Перед изменением файла необходимо понять:

что именно работает;

что именно сломано;

что именно требуется изменить.

Изменение "на всякий случай"

запрещено.

---

# 9. Документация

Если принято важное решение,

оно должно быть отражено в:

ADOKVA_MASTER.md

или

PROJECT_STRUCTURE.md

или

DEVELOPMENT_RULES.md

---

# 10. Проверка сайта

После изменения обязательно проверить:

• запускается ли сайт;

• работает ли поиск;

• работают ли карточки;

• работает ли Земля;

• работают ли перелёты;

• работает ли управление камерой.

---

# 11. Общение

Если пользователь показывает ошибку —

сначала анализ.

Потом исправление.

Не делать предположений без проверки.

---

# 12. Главная цель

Разрабатывать ADOKVA постепенно.

Сохранять рабочее состояние проекта.

Не жертвовать стабильностью ради скорости.

---
