# PROJECT RULES

Версия документа: 1.0

---

# Назначение

Этот документ определяет единые правила разработки проекта ADOKVA.

Все новые компоненты, системы и функции должны соответствовать данным правилам.

Нарушение правил допускается только после принятия нового архитектурного решения и обязательного обновления документации.

---

# Главное правило

Архитектура важнее скорости разработки.

Лучше потратить больше времени на правильную структуру, чем быстро написать код, который придётся переписывать.

---

# Основные принципы

Проект должен быть:

• понятным;

• предсказуемым;

• масштабируемым;

• легко поддерживаемым;

• удобным для развития.

---

# Один компонент — одна задача

Каждый компонент отвечает только за одну область ответственности.

Примеры:

✔ Earth отображает Землю.

✔ Sun отображает Солнце.

✔ Mercury отображает Меркурий.

✔ SearchPanel отвечает только за поиск.

✔ WorldInfoPanel отвечает только за отображение информации.

Компоненты не должны выполнять чужую работу.

---

# Один файл — одна ответственность

Файл не должен одновременно:

- рисовать интерфейс;
- выполнять вычисления;
- хранить данные;
- управлять камерой.

Каждая задача выносится в отдельный файл.

---

# Разделение данных и логики

Данные никогда не смешиваются с визуальными компонентами.

Правильно:

```
data/
    worldInformation.ts

components/
    WorldInfoPanel.tsx
```

Неправильно:

```
WorldInfoPanel.tsx

↓

1000 строк данных внутри компонента
```

---

# Константы

Любые повторяющиеся значения выносятся в отдельные константы.

Например:

```
SUN_POSITION

EARTH_RADIUS

CAMERA_SPEED

DEFAULT_ZOOM
```

Не допускается использование "магических" чисел.

---

# Типизация

Все пользовательские структуры описываются через TypeScript.

Использование `any` допускается только как временная мера и должно быть устранено.

---

# Папки

Каждая папка имеет собственную ответственность.

Нельзя размещать файлы случайным образом.

Перед созданием нового файла необходимо определить его место в архитектуре.

---

# Именование

Используются понятные английские названия.

Компоненты:

```
Earth.tsx

CameraController.tsx

WorldInfoPanel.tsx
```

Функции:

```
moveCamera()

focusPlanet()

calculateOrbit()
```

Переменные:

```
selectedWorld

cameraTarget

orbitRadius
```

---

# Комментарии

Код должен быть понятен без большого количества комментариев.

Комментарии используются только там, где логика действительно сложная или неочевидная.

---

# Повторное использование

Если одинаковый код появляется дважды — необходимо подумать о его вынесении в отдельную функцию, хук или компонент.

---

# Документация

Любое изменение архитектуры сопровождается обновлением документации.

Минимальный список:

- CHANGELOG.md
- CURRENT_STATE.md
- DECISIONS.md

При необходимости обновляются и другие документы.

---

# Работа с ошибками

Если обнаружена ошибка:

1. Найти причину.
2. Исправить её.
3. Проверить, что исправление не повлияло на другие системы.
4. Зафиксировать решение в BUG_HISTORY.md, если ошибка архитектурно важна.

---

# Работа с новыми функциями

Каждая новая функция проходит следующие этапы:

1. Анализ.
2. Проектирование.
3. Определение архитектуры.
4. Реализация.
5. Проверка.
6. Обновление документации.

---

# Git

После завершения значимого этапа рекомендуется создать отдельный коммит.

Сообщения коммитов должны быть понятными и отражать выполненную работу.

---

# Производительность

При добавлении новых возможностей необходимо учитывать:

- количество объектов в сцене;
- количество перерисовок;
- использование памяти;
- плавность анимации;
- удобство дальнейшего масштабирования.

---

# Будущие расширения

Каждая новая система должна подключаться без переписывания существующих компонентов.

Архитектура должна быть открыта для расширения и закрыта для необоснованных изменений.

---

# Финальное правило

Каждое решение должно отвечать на вопрос:

"Станет ли проект проще поддерживать через год после этого изменения?"

Если ответ отрицательный — решение необходимо пересмотреть.

---

Конец документа.
