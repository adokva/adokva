# ARCHITECTURE

Версия документа: 1.0

---

# Назначение документа

Данный документ описывает архитектуру проекта ADOKVA.

Его цель — объяснить не только структуру каталогов, но и принципы взаимодействия между всеми подсистемами проекта.

Архитектура должна оставаться понятной даже спустя годы развития проекта.

---

# Основная идея

ADOKVA строится как модульная система.

Каждая подсистема отвечает только за свою область ответственности.

Подсистемы взаимодействуют друг с другом через понятные интерфейсы и не должны содержать скрытых зависимостей.

---

# Главные архитектурные принципы

• Один компонент — одна ответственность.

• Один файл — одна задача.

• Минимальная связанность между компонентами.

• Максимальная переиспользуемость.

• Простая навигация по проекту.

• Предсказуемое расположение файлов.

---

# Общая структура проекта

```
app/
components/
data/
hooks/
lib/
public/
styles/
types/
utils/
docs/
```

Каждая папка имеет строго определённую задачу.

---

# app/

Содержит страницы приложения.

Здесь практически не должно быть сложной логики.

Главная задача папки — собрать приложение из готовых компонентов.

Здесь размещаются:

- layout
- page
- глобальные настройки

---

# components/

Самая большая часть проекта.

Каждый визуальный объект располагается в отдельном компоненте.

Например:

```
Earth.tsx

Moon.tsx

Sun.tsx

Mercury.tsx

Mars.tsx

CameraController.tsx

WorldInfoPanel.tsx

SearchPanel.tsx

Menu.tsx
```

Компоненты не должны зависеть друг от друга напрямую.

---

# data/

Хранит исключительно данные.

Например:

```
WORLD_INFORMATION

PLANETS

COUNTRIES

CITIES

LANGUAGES
```

Никакой логики.

Только структуры данных.

---

# hooks/

Повторно используемая логика React.

Например:

```
useCamera()

usePlanet()

useSearch()

useAnimation()

useSelection()
```

Любой хук отвечает только за одну систему.

---

# lib/

Низкоуровневая логика проекта.

Например:

```
space.ts

camera.ts

math.ts

constants.ts
```

Здесь располагаются вычисления, константы и общие функции.

---

# utils/

Вспомогательные функции.

Например:

```
degToRad()

latLngToVector3()

clamp()

lerp()

formatDistance()
```

Utils не должны зависеть от компонентов.

---

# public/

Ресурсы проекта.

Текстуры.

Иконки.

Изображения.

Шрифты.

Звуки.

---

# styles/

Глобальные стили.

Темы.

Переменные.

Общие CSS-модули.

---

# types/

Все пользовательские типы TypeScript.

Например:

```
Planet

Country

City

CameraTarget

SpaceObject
```

Все типы централизованы.

---

# Поток данных

Общий принцип движения данных:

```
Data

↓

Logic

↓

Components

↓

UI

↓

User
```

Данные всегда идут сверху вниз.

Компоненты не должны самостоятельно изменять исходные данные.

---

# Поток выбора объекта

```
Пользователь

↓

Клик по объекту

↓

Определение объекта

↓

Обновление состояния

↓

Перемещение камеры

↓

Открытие панели информации
```

Каждый этап независим.

---

# Добавление новой планеты

Добавление нового объекта должно состоять из нескольких простых шагов.

1.

Создать компонент.

2.

Добавить позицию.

3.

Добавить данные.

4.

Добавить информационную карточку.

5.

Подключить выбор.

6.

Подключить камеру.

Никаких изменений существующей архитектуры происходить не должно.

---

# Масштабируемость

Архитектура должна позволять безболезненно добавлять:

- новые планеты;

- спутники;

- страны;

- города;

- маршруты;

- погодные сервисы;

- спутниковые данные;

- AI;

- внешние API.

Без переписывания уже существующего кода.

---

# Что запрещается

Запрещается:

• создавать огромные компоненты;

• дублировать код;

• смешивать данные и интерфейс;

• хранить бизнес-логику внутри JSX;

• использовать "магические" числа без констант;

• нарушать структуру каталогов.

# Принцип качества

Рабочий код — это только половина задачи.

Вторая половина — понятная архитектура.

Если решение сложно объяснить, значит архитектуру необходимо улучшить.

---

# Архитектурное правило ADOKVA

Ни одна новая функция не должна ухудшать существующую структуру проекта.

Если для добавления новой возможности приходится переписывать множество существующих компонентов, значит архитектурное решение требует пересмотра.

Архитектура должна помогать развитию проекта, а не препятствовать ему.
