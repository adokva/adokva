# DECISIONS

## Архитектурные решения проекта ADOKVA

Этот документ хранит не только принятые решения, но и причины, по которым они были приняты.

---

# Решение №1

## Новый старт проекта

Первоначальный проект был остановлен.

Было принято решение создать новый проект `adokva2` с чистой архитектурой.

### Причина

Исправлять накопившиеся проблемы оказалось сложнее, чем начать с понятной структуры.

---

# Решение №2

## Правило ЙЙ

Все изменения выполняются по одному файлу.

Порядок работы:

1. Полная замена содержимого файла.
2. Сохранение (`Ctrl + S`).
3. Проверка `Problems`.
4. Только после этого переход к следующему файлу.

### Причина

Такой подход позволяет быстро находить ошибки и не ломать уже работающий код.

---

# Решение №3

## Модульная камера

Логика камеры разделена на отдельные компоненты.

Основные модули:

- CameraIntro
- FlightCamera
- FocusOrbitControls
- WorldCameraController

Дополнительные математические модули находятся в папке `components/camera`.

### Причина

Каждый компонент отвечает только за одну задачу, что упрощает поддержку и развитие проекта.

---

# Решение №4

## Постепенный рефакторинг

Работающий код не переписывается полностью.

Любые изменения выполняются небольшими безопасными шагами.

### Причина

Это снижает риск появления трудноуловимых ошибок.

---

# Решение №5

## Документация

Каждый важный этап сопровождается обновлением документации.

### Причина

Проект должен быть понятен даже после длительного перерыва в разработке.

---

# Основной принцип

Перед добавлением новой функции задаётся вопрос:

> Сделает ли это проект стабильнее, понятнее и удобнее?

Если ответ отрицательный — решение пересматривается.
